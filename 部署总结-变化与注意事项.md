## 二、核心思想

### 1. 前后端分离部署

- **前端**：静态资源（HTML/CSS/JS）→ 部署到 Azure Static Web Apps。
- **后端**：API 服务 → 已在 Azure（如 Function App / App Service）。
- **关系**：前端在「构建时」必须知道后端的完整 URL，打包进前端代码。

### 2. 环境区分

- **开发环境**：`npm run dev`，用 Vite 代理把 `/api` 转到后端（可指向本地或 Azure）。
- **生产环境**：`npm run build` 时通过 `VITE_API_BASE_URL` 把后端地址写进打包结果；部署后浏览器直接请求该 URL，不再经过代理。

### 3. 构建时注入 > 运行时配置

- Vite 只会在 **构建时** 读取 `VITE_*` 环境变量并替换到代码里。
- 所以：**正确的后端地址必须在「构建」那一步就设好**（例如在 GitHub Actions 的 `env` 里），不能只依赖「部署后在 Azure 里改设置」。

## 三、反复犯的错误与注意事项

### 错误 1：后端地址写错或写短

- **现象**：页面一直「加载中」或空白，控制台 `net::ERR_NAME_NOT_RESOLVED`。
- **原因**：用了不存在的域名（如 `toutiao-backend.azurewebsites.net`），或漏掉一段（如少了 `canadacentral-01`）。
- **注意**：
  - 后端 URL 必须从 Azure 后台/Function 的「真实访问地址」完整复制。
  - 包含 `https://`、完整子域名、区域（如 `canadacentral-01`）等，不要手改或「猜」。

### 错误 2：本地以为有本地后端（127.0.0.1:8000）

- **现象**：本地 3000 打开时，控制台 `ECONNREFUSED 127.0.0.1:8000`。
- **原因**：Vite 代理指向本地 8000，但本机没跑后端。
- **注意**：若后端只在 Azure，本地开发时代理应指向 **Azure 后端 URL**，而不是 `127.0.0.1:8000`。

### 错误 3：以为改 Azure 门户里的「应用设置」就能改前端里的 API 地址

- **现象**：在 Static Web App 里加了 `VITE_API_BASE_URL`，但页面仍然请求错误地址。
- **原因**：Vite 在 **build** 时就把 `import.meta.env.VITE_API_BASE_URL` 替换成常量了，部署后的「应用设置」不会改变已打包的 JS。
- **注意**：生产环境的 API 地址必须在 **GitHub Actions 的 build 步骤**（或你执行 `npm run build` 的环境）里通过 `env.VITE_API_BASE_URL` 传入，并重新构建、再部署。

### 错误 4：忘记 CORS

- **现象**：部署后前端能打开，但接口报 CORS 错误。
- **注意**：后端必须配置 CORS，允许前端域名（如 `https://xxx.azurestaticapps.net`）和本地（如 `http://localhost:3000`）。

## 四、部署的总体思想

1. **后端先就绪**：API 已在 Azure 上并可访问（有正确 URL）。
2. **前端「唯一依赖」**：知道这个后端 URL，并在 **构建时** 写进前端。
3. **构建与部署分离**：
   - 构建：在 CI（如 GitHub Actions）或本机执行 `npm run build`，传入正确的 `VITE_API_BASE_URL`。
   - 部署：把生成的 `dist` 交给 Azure Static Web Apps（或上传/通过 GitHub 自动部署）。
4. **环境一致**：本地开发、CI 构建、线上运行用的「后端地址」要一致或按环境区分清楚，避免混用错误域名。

---

## 五、大致步骤（清单）

| 步骤 | 内容 |
|------|------|
| 1 | 确认后端已在 Azure 运行，并拿到**完整** API 根地址（如 `https://xxx.canadacentral-01.azurewebsites.net`）。 |
| 2 | 前端用环境变量：生产环境用 `VITE_API_BASE_URL`，在 `src/config/api.js` 中读取。 |
| 3 | 本地开发：在 `vite.config.js` 的 proxy 里把 `/api` 指到后端（本地或 Azure）。 |
| 4 | 构建：执行 `npm run build`，且**必须**在构建环境中设置 `VITE_API_BASE_URL` 为正确后端地址。 |
| 5 | 部署：把 `dist` 部署到 Azure Static Web Apps（如通过 GitHub Actions 推代码自动构建+部署）。 |
| 6 | CI 配置：在 `.github/workflows/azure-static-web-apps.yml` 的 build 步骤里设置 `env.VITE_API_BASE_URL`，不要写错、写短。 |
| 7 | 后端 CORS：允许前端域名和本地开发地址。 |
| 8 | 验证：打开线上前端 URL，看接口是否请求到正确后端、是否有 CORS 报错。 |

---

## 六、一句话记忆

- **后端地址要在「构建」时写对，并完整复制 Azure 给的 URL；本地没后端就把代理指到 Azure；部署后仍空白先查是不是 URL 错了（ERR_NAME_NOT_RESOLVED）。**
